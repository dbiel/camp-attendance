DATABASE SCHEMA
================================================================================

Tables (7):
  1. students (644 capacity)
     - Full name, preferred name, gender
     - Instrument, ensemble, chair number
     - Division (commuter/overnight)
     - Contact info (email, phone)
     - Dorm assignment
     - Parent contact info
     - Medical notes, additional info

  2. faculty (88 capacity)
     - Name, role
     - Email

  3. periods (11, pre-seeded)
     - Number, name
     - Start/end times

  4. sessions
     - Period, name, type
     - Location, faculty assignment
     - Ensemble/instrument assignment

  5. session_students (many-to-many)
     - Maps students to sessions
     - Unique constraint on (session_id, student_id)

  6. attendance
     - Student, session, date
     - Status (present/absent/tardy)
     - Marked by, marked at
     - Unique constraint on (student_id, session_id, date)

  7. schedule_templates (optional)
     - For future auto-assignment features

Indexes (10):
  - Students by ensemble, instrument, dorm
  - Sessions by period, faculty
  - Session-students by session, student
  - Attendance by student, session, date

Constraints:
  - Foreign keys on all relationships
  - Check constraints on status values
  - Check constraint on division values
  - Unique constraints for data integrity

Faculty:
  GET    /api/faculty               - List all faculty
  POST   /api/faculty               - Create faculty member
  GET    /api/faculty/[id]          - Get faculty member
  PUT    /api/faculty/[id]          - Update faculty member
  DELETE /api/faculty/[id]          - Delete faculty member
  GET    /api/faculty/[id]/sessions - Get faculty's sessions

Sessions:
  GET    /api/sessions              - List all sessions

  PUT    /api/sessions/[id]         - Update session
  DELETE /api/sessions/[id]         - Delete session
  GET    /api/sessions/[id]/students - Get students in session

Attendance:
  GET    /api/attendance?session_id=&date= - Get attendance
